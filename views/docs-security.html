Security Documentation

Two-factor authentication, password policies, session management, and audit logging for the NetSudo platform.

Two-Factor Authentication (2FA) Setup

NetSudo supports TOTP-based two-factor authentication using the speakeasy library. Once enabled, users must enter a 6-digit code from their authenticator app after entering their password.

Compatible Apps

  • Google Authenticator (iOS / Android)
  • Authy (iOS / Android / Desktop)
  • Microsoft Authenticator (iOS / Android)
  • 1Password (built-in TOTP support)
  • Any app that supports standard TOTP (RFC 6238)

Enabling 2FA — Step by Step

  1. Go to Settings
    Navigate to /admin/settings from the sidebar or go directly to the Settings page.
  2. Open the 2FA section
    Scroll to the Two-Factor Authentication section. You'll see the current status (Disabled/Enabled).
  3. Click "Set Up 2FA"
    The system generates a unique TOTP secret and displays a QR code. This QR code contains your secret key encoded as an otpauth:// URL.
  4. Scan the QR code
    Open your authenticator app and scan the QR code. The app will create a new entry labeled "NetSudo (yourusername)" that generates 6-digit codes every 30 seconds.
  5. Enter the verification code
    Type the current 6-digit code from your authenticator app into the verification field. This confirms you've successfully set up the app.
  6. Save your backup secret
    The text-based secret key is also displayed below the QR code (e.g., JBSWY3DPEHPK3PXP). Save this somewhere secure — you'll need it if you lose access to your authenticator app.
  7. 2FA is now active
    On your next login, after entering your password, you'll be redirected to the 2FA page where you'll enter the current 6-digit code.
Save Your Secret Key If you lose access to your authenticator app and don't have your backup secret key, you will need a superadmin to reset your 2FA by directly updating the database. Store your secret key in a password manager or write it down securely.

Login Flow with 2FA

  1. Enter username and password
    Go to /admin/login and enter your credentials as usual.
  2. Redirected to 2FA page
    If your password is correct and 2FA is enabled, you're redirected to /admin/2fa.
  3. Enter your 6-digit code
    Open your authenticator app and enter the current code. Codes refresh every 30 seconds. There is a 1-code window tolerance (the previous code is also accepted).
  4. Access granted
    On success, you're redirected to the dashboard with a full session.
Failed 2FA Attempts After 3 consecutive wrong 2FA codes, your session is destroyed and you're redirected back to the login page. You must re-enter your username and password. This prevents brute-force TOTP attacks.

Disabling 2FA

To disable 2FA:

  1. Go to /admin/settings
  2. Click "Disable 2FA"
  3. Enter your current password to confirm
  4. 2FA is removed — future logins will only require username and password

2FA API Endpoints

MethodPathDescription
GET/admin/settings/2faGet 2FA status. If disabled, returns QR code and secret for setup.
POST/admin/settings/2fa/enableEnable 2FA by submitting a valid TOTP code. Requires a pending secret in session.
POST/admin/settings/2fa/disableDisable 2FA. Requires current password in request body.

2FA API Example

bash 
# Step 1: Get 2FA setup info (QR code + secret)
curl https://netsudo.com/admin/settings/2fa \
  -b cookies.txt
# Response (2FA not yet enabled):
# {
#   "enabled": false,
#   "secret": "JBSWY3DPEHPK3PXP",
#   "qr": "data:image/png;base64,..."
# }

# Step 2: Enable 2FA with a valid TOTP code
curl -X POST https://netsudo.com/admin/settings/2fa/enable \
  -H "Content-Type: application/json" \
  -b cookies.txt \
  -d '{"token": "123456"}'
# Response: {"success": true}

# Disable 2FA (requires password)
curl -X POST https://netsudo.com/admin/settings/2fa/disable \
  -H "Content-Type: application/json" \
  -b cookies.txt \
  -d '{"password": "YourCurrentPassword"}'
# Response: {"success": true}

Password Policy

NetSudo enforces the following password requirements:

RuleRequirement
Minimum length8 characters
Hashing algorithmbcrypt with 10 salt rounds
StorageOnly the hash is stored — plaintext is never saved
Case sensitivityPasswords are case-sensitive

Changing Your Password

  1. Navigate to Settings
    Go to /admin/settings from the sidebar.
  2. Enter your current password
    This verifies your identity before allowing a password change.
  3. Enter your new password
    Must be at least 8 characters. Use a strong, unique password that you don't use elsewhere.
  4. Submit
    Your password is immediately updated. Your current session remains active.

Best Practices

  • Use a password manager (1Password, Bitwarden, etc.) to generate and store strong passwords
  • Use at least 12+ characters with a mix of uppercase, lowercase, numbers, and symbols
  • Never reuse passwords across different services
  • Enable 2FA in addition to a strong password for maximum security
  • Change your password immediately if you suspect it has been compromised

Password Reset (Admin)

There is no self-service password reset. If you forget your password:

  • Contact a superadmin who can reset your password via the Users management page (/admin/users)
  • The superadmin can update your password through PUT /admin/api/users/:id
  • After reset, immediately change to your own password via Settings

Session Management

NetSudo uses server-side sessions via express-session. Sessions are stored in server memory (not in cookies or the database).

Session Cookie Configuration

PropertyValuePurpose
httpOnly true Prevents JavaScript from reading the session cookie, protecting against XSS-based session theft
secure true Cookie is only transmitted over HTTPS — never sent over plain HTTP
sameSite strict Cookie is never sent with cross-origin requests, preventing CSRF attacks
maxAge 8 hours Session automatically expires after 8 hours, requiring re-authentication

Session Lifecycle

  1. Login — A new session is created with the user's id, username, and role
  2. 2FA (if enabled) — During 2FA verification, the user data is stored as pendingUser until the TOTP code is verified
  3. Active — Session is valid for all authenticated requests. The session cookie (connect.sid) is sent with every request
  4. Expiry — After 8 hours of inactivity, the session expires automatically
  5. Logout — Calling GET /admin/logout destroys the session immediately

Security Implications

  • No session persistence — If the server restarts, all sessions are destroyed and users must log in again
  • Single server — Server-side sessions are stored in memory, so they don't work across multiple server instances without a shared session store
  • Cookie theft protection — The combination of httpOnly, secure, and sameSite: strict makes session hijacking extremely difficult
  • No remember-me — There is no "remember me" option; all sessions expire after 8 hours
Trust Proxy The server has trust proxy enabled (app.set('trust proxy', 1)) to correctly handle the client's IP address when behind Nginx reverse proxy. This ensures rate limiting and audit logs capture the real client IP, not the proxy IP.

Audit Log

NetSudo maintains two types of audit logs: login attempts and activity logs.

Login Attempts Log

Every login attempt (successful or failed) is recorded in the login_attempts table.

FieldTypeDescription
idIntegerAuto-incrementing primary key
usernameStringUsername attempted (may not be a valid user)
ipStringClient IP address (respects X-Forwarded-For)
successBooleantrue for successful login, false for failed
user_agentStringBrowser/client user agent string
created_atTimestampWhen the attempt occurred

Viewing Login Attempts

Superadmins can view the last 100 login attempts via:

bash 
curl https://netsudo.com/admin/api/login-attempts \
  -b cookies.txt

# Example response:
# [
#   {
#     "id": 147,
#     "username": "admin",
#     "ip": "203.0.113.42",
#     "success": true,
#     "user_agent": "Mozilla/5.0 ...",
#     "created_at": "2026-03-06T14:32:00.000Z"
#   },
#   {
#     "id": 146,
#     "username": "admin",
#     "ip": "198.51.100.7",
#     "success": false,
#     "user_agent": "curl/8.0",
#     "created_at": "2026-03-06T13:10:00.000Z"
#   }
# ]

What to Look For

  • Multiple failed attempts from one IP — Possible brute-force attack. The account lockout mechanism handles this automatically after 5 failures
  • Failed attempts for unknown usernames — Someone is guessing usernames. Consider IP blocking at the Nginx level
  • Successful logins from unfamiliar IPs — Could indicate credential compromise. Verify with the user and consider requiring 2FA
  • Unusual user agents — Automated tools (curl, python-requests) hitting the login endpoint may indicate an attack
  • Login outside business hours — Compare timestamps against expected usage patterns

Rate Limiting

NetSudo uses express-rate-limit to prevent brute-force attacks on the login endpoint.

SettingValue
Window15 minutes
Max requests10 per window per IP
Applies toPOST /admin/login
HeadersStandard rate-limit headers (RateLimit-*)
Response when limited429 with {"error": "Too many login attempts. Try again in 15 minutes."}

Account Lockout

In addition to IP-based rate limiting, NetSudo implements per-account lockout:

  1. Each failed login increments the user's failed_attempts counter
  2. After 5 consecutive failures, the account is locked for 15 minutes
  3. A Telegram notification is sent to the admin with the username and IP address
  4. The lockout message shows the remaining minutes until the account unlocks
  5. A successful login resets the counter to 0 and clears any lockout
Dual Protection Rate limiting protects against rapid brute-force from a single IP. Account lockout protects against distributed attacks across multiple IPs targeting the same account. Together, they provide comprehensive protection.

Content Security Policy & Headers

NetSudo uses Helmet.js to set security headers on all responses. Helmet configures multiple HTTP headers that protect against common web vulnerabilities.

CSP Directives

DirectiveValuePurpose
default-src 'self' Only allow resources from the same origin by default
style-src 'self', 'unsafe-inline', Google Fonts, cdnjs Allow styles from self, inline styles, Google Fonts, and cdnjs (Font Awesome)
font-src 'self', Google Fonts Allow fonts from self and Google Fonts CDN
script-src 'self', 'unsafe-inline', cdnjs Allow scripts from self, inline scripts, and cdnjs (SortableJS, etc.)
img-src 'self', data:, https: Allow images from self, data URIs (QR codes), and any HTTPS source
connect-src 'self' Only allow API calls to the same origin

Other Security Headers

Helmet also sets these headers automatically:

  • X-Content-Type-Options: nosniff — Prevents MIME type sniffing
  • X-Frame-Options: SAMEORIGIN — Prevents clickjacking via iframes
  • X-XSS-Protection — Legacy XSS protection header
  • Strict-Transport-Security — Enforces HTTPS connections
  • X-DNS-Prefetch-Control: off — Disables DNS prefetching
  • X-Download-Options: noopen — Prevents IE from opening downloads directly
  • X-Permitted-Cross-Domain-Policies: none — Blocks Flash/PDF cross-domain access
  • Referrer-Policy: no-referrer — Prevents referrer leakage
